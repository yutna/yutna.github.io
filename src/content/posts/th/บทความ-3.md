  จนถึงตอนนี้ การพยายามที่จะจัดรูปแบบบทความ เอกสาร หรือโพสต์บล็อกด้วย Tailwind
  ได้เป็นงานที่ลำบากที่ต้องใช้สายตาที่คมและ CSS ที่ซับซ้อนมากมาย

ตามค่าเริ่มต้น Tailwind จะลบการจัดรูปแบบเริ่มต้นจากเบราว์เซอร์ทั้งหมดออกจากย่อหน้า
หัวข้อ รายการ และอื่นๆทำให้มีประโยชน์มากในการสร้าง UI แอปพลิเคชันเนื่องจากคุณใช้
เวลาน้อยกว่าในการยกเลิกสไตล์ของ user-agent แต่เมื่อคุณ _จริงๆ_ แล้วต้องการจัดรูป
แบบบางเนื้อหาที่มาจาก Rich-text editor ใน CMS หรือไฟล์ markdown มันสามารถทำให้แปลกและไม่ถูกต้องได้

เราได้รับคำบรรยายมากมายเกี่ยวกับนี้จริงๆ โดยที่คนบ่อยครั้งถามเราเรื่องเช่น:

> ทำไม Tailwind ถึงลบสไตล์เริ่มต้นของ `h1` ออก?
ฉันจะทำอย่างไรเพื่อปิดการใช้งานนี้?
และคุณหมายถึงว่าฉันจะสูญเสียสไตล์เริ่มต้นอื่นๆ ไปทั้งหมดด้วยหรือไม่?

เรารับรู้ความเห็นของคุณ แต่เราไม่เชื่อว่าการปิดใช้งานสไตล์เริ่มต้นของเราเพียงอย่างเดียวคือสิ่งที่คุณต้องการจริงๆ
คุณไม่ต้องการต้องลบระยะขอบที่น่ารำคาญทุกครั้งที่คุณใช้ `p` ในส่วนหนึ่งของ UI ของแดชบอร์ดของคุณ
และฉันเสมอแล้วว่าคุณไม่จริงจังที่จะให้โพสต์บล็อกของคุณใช้สไตล์ของ user-agent
คุณต้องการให้มันดู _น่าทึ่ง_ ไม่ใช่น่าเบื่อ

ปลั๊กอิน `@tailwindcss/typography` เป็นพยานที่เราพยายามทำให้คุณได้รับสิ่งที่คุณ
_ต้องการจริงๆ_ โดยไม่มีข้อเสียจากการทำบางสิ่งที่โง่โดยการปิดใช้งานสไตล์เริ่มต้นของเรา

มันเพิ่มคลาส `prose` ใหม่ที่คุณสามารถใส่ในบล็อกของเนื้อหา HTML ปกติและทำให้มันกลายเป็นเอกสารที่สวยงาม
และจัดรูปแบบได้ดี:

```html
<article class="prose">
  <h1>ขนมปังกระเทียมกับชีส: สิ่งที่วิทยาศาสตร์บอกให้เราทราบ</h1>
  <p>
    มาหลายปีแล้วที่พ่อแม่ได้เผยแพร่ประโยชน์ทางสุขภาพของการรับประทานขนมปังกระเทียมกับชีสให้กับลูกของพวกเขา
    อาหารนี้ได้รับสถานะที่โดดเด่นถึงขั้นที่เป็นไอคอนในวัฒนธรรมของเรา ทำให้เด็กๆ บางครั้งอาจแต่งตัวเป็นขนมปังที่อบอุ่นและมีชีสในงานฮาโลวีน
  </p>
  <p>
    แต่การศึกษาล่าสุดแสดงให้เห็นว่าอาหารว่างที่ได้รับความนิยมนี้อาจเกี่ยวข้องกับกรณีของโรคพิษสุนัขที่เกิดขึ้นทั่วประเทศ
  </p>
  <!-- ... -->
</article>
```

สำหรับข้อมูลเพิ่มเติมเกี่ยวกับวิธีการใช้ปลั๊กอินและคุณลักษณะที่รวมอยู่ [อ่านเอกสารประกอบ](https://github.com/tailwindcss/typography/blob/master/README.md).

## คาดหวังว่าจะเกิดอะไรต่อไปบ้าง

สิ่งที่ตามมาจากที่นี่คือเพียงชุดของข้อความที่ไม่มีสาระทั้งหลายที่ฉันเขียนขึ้น
เพื่อทดสอบปลั๊กอินเอง มันประกอบด้วยทุกรูปแบบข้อความที่มีสาระที่ฉันคิดออกมา เช่น
**ข้อความหนา** รายการที่ไม่มีลำดับ รายการที่มีลำดับ บล็อกโค้ด บล็อกคำพูด
_และแม้กระทั่งตัวเอียง_ ด้วย

การครอบคลุมทุกรูปแบบการใช้งานนี้มีความสำคัญเพราะเหตุผลหลายประการ:

1. เราต้องการให้ทุกสิ่งดูดีทันทีที่ถูกใช้งาน
2. จริงๆ แล้วเป็นเหตุผลแรกเพียงอย่างเดียวนั้นคือจุดมุ่งหลักของปลั๊กอิน
3. นี่คือเหตุผลที่สามที่เสมือนจริงแม้ว่ารายการที่มีสามรายการจะดูเหมือนจริงได้มากกว่ารายการที่มีสองรายการ

ตอนนี้เราจะลองใช้รูปแบบของหัวข้ออีกแบบ

- ดังนั้นนี่คือรายการแรกในรายการนี้

## ถ้าเราเรียงหัวข้อเข้าด้วยกันจะเป็นอย่างไร?

### เราควรแน่ใจว่ามันดูดีด้วยเช่นกัน

บางครั้งคุณมีหัวข้อติดกันโดยตรง ในกรณีนั้น คุณต้องบางครั้งยกเลิกการใช้ margin ด้านบนของหัวข้อที่สองเพราะมันมักจะดูดีกว่าหัวข้อที่ต้องการจะให้มีระยะห่างกันน้อยกว่าย่อหน้าที่ตามหลังหัวข้อควรจะมี

### เมื่อหัวข้อตามหลังย่อหน้า …

เมื่อหัวข้อตามหลังย่อหน้า เราต้องการช่องว่างเพิ่มเติม ตามที่ฉันได้กล่าวถึงไว้ข้าง
บนแล้ว ตอนนี้เราจะดูรายการที่ซับซ้อนมากขึ้นกันบ้าง

- **ฉันมักทำสิ่งนี้บ่อยครั้งเมื่อรายการมีหัวข้อ**

  ด้วยเหตุผลที่หนึ่งฉันคิดว่ามันดูเท่ห์ ซึ่งน่าเสียดายเพราะมันค่อนข้างยุ่งยากที่จะทำให้รูปแบบดูถูกต้อง

  ฉันมักมีข้อความที่มีสองหรือสามย่อหน้าในรายการนี้ด้วย ดังนั้นส่วนที่ยากคือการทำ
  ให้ระยะห่างระหว่างย่อหน้า หัวข้อของรายการ และรายการที่แยกกันทั้งหมด
  มีความสัมพันธ์ที่เหมาะสม ค่อนข้างยากอย่างแท้จริง คุณสามารถทำสาระที่ว่าคุณควรจะไม่เขียนแบบนี้

- **เนื่องจากรายการนี้ ฉันต้องการอย่างน้อยสองรายการ**

  ฉันได้อธิบายว่าฉันกำลังทำอะไรในรายการก่อนหน้านี้แล้ว แต่รายการจะไม่เป็นรายการ
  หากมีเพียงรายการเดียว และเราต้องการให้มันดูเหมือนจริง นั่นเป็นเหตุผลที่ฉันเพิ่มรายการที่สองนี้เพื่อให้ฉันมีสิ่งที่จะดูเมื่อเขียนสไตล์

- **ไม่เลวเลยที่จะเพิ่มรายการที่สามด้วย**

  คิดว่ามันคงไม่เป็นไรถ้าใช้สองรายการ แต่สามรายการแน่นอนไม่เป็นห่วง และเนื่องจาก
  ฉันดูเหมือนจะไม่มีปัญหาในการทำสิ่งที่ไม่มีเหตุผลที่จะพิมพ์ ผมคิดว่าควรรวมมัน
  เข้าด้วยกัน ผมจะกด <kbd>Enter</kbd> ตอนนี้

หลังจากรายการแบบนี้ ฉันโดยปกติจะมีคำพูดหรือย่อหน้าส่วนปิด เพราะมันดูเหมือนแปลกที่จะไปทันทีสู่หัวข้อต่อไป

## โค้ดควรดูดีโดยค่าเริ่มต้น

คิดว่าคนส่วนใหญ่จะใช้ [highlight.js](https://highlightjs.org/) หรือ
[Prism](https://prismjs.com/) หรือบางอย่างถ้าพวกเขาต้องการจัดรูปแบบบล็อกโค้ด
ของพวกเขา แต่ไม่เป็นไรที่จะทำให้ดู _ดี_ ทันทีที่ถูกใช้ โดยไม่ต้องมีการเน้นไวยากรณ์

นี่คือรูปแบบของไฟล์ `tailwind.config.js` ที่เป็นค่าเริ่มต้นในขณะที่เขียนข้อความนี้:

```js
module.exports = {
  purge: [],
  theme: {
    extend: {},
  },
  variants: {},
  plugins: [],
}
```

หวังว่ามันดูดีมีความพอใจสำหรับคุณ

### และเรื่องของรายการที่ซ้อนกัน?

รายการที่ซ้อนกันโดยทั่วไปมีลักษณะที่ไม่ดี นั่นคือเหตุผลที่บางตัวแก้ไขเช่น Medium
ไม่อนุญาตให้คุณทำ แต่คิดว่าหากบางคุณยังคงทำ เราต้องหนักหน่วงในการทำให้มันทำงานอย่างน้อยละเอียด

1. **การใช้รายการที่ซ้อนกันนั้นหากแย่มากน้อยเสมอ**
    - คุณอาจรู้สึกว่าคุณกำลังเป็น "จัดระเบียบ" หรืออะไรทำนองนี้ แต่คุณกำลังสร้างรูปร่างที่ไม่สวยบนหน้าจอที่ยากต่อการอ่าน
    - การนำทางที่ซ้อนกันใน UI ก็ไม่ดีนัก ควรเรียงสิ่งที่เกี่ยวข้องให้มีโครงสร้างแบบแผ่นเสมอไว้ที่สุด
    - การซ้อนกันของโฟลเดอร์มากมายในโค้ดของคุณก็ไม่เป็นประโยชน์
2. **เนื่องจากเราต้องการมีรายการเพิ่มเติม นี่คืออีกหนึ่งรายการ**
    - ฉันไม่แน่ใจว่าเราจะทำให้มีสไตล์มากกว่าสองระดับลึก
    - สองระดับลึกนั้นเป็นมากพอแล้ว สามระดับลึกมักจะเป็นไอเดียที่ไม่ดี
    - ถ้าคุณซ้อนกันถึงระดับที่สี่ลึก คุณควรอยู่ในโรงจำ (ท่าทีสามารถเป็นการใช้รูปพรรณแบบเต็มรูปแบบ)
3. **สองรายการไม่ได้เป็นรายการจริงๆ แต่สามรายการนั้นดีลง**
    - อีกครั้งๆ กรุณาอย่าซ้อนรายการถ้าคุณต้องการให้คนอ่านเนื้อหาของคุณให้จริงจัง
    - ไม่มีคนต้องการมองที่นี้
    - ฉันรู้สึกไม่พอใจที่เราต้องให้ความสนใจถึงการจัดรูปแบบนี้

สิ่งที่น่ารำคาญที่สุดเกี่ยวกับรายการใน Markdown คือ `<li>` elements ไม่ได้รับ
`<p>` tag ลูกๆ นอกจากกรณีที่มีย่อหน้ามากกว่าหนึ่งในรายการนั้นๆ นั่นหมายความว่าฉันต้องกังวลเกี่ยวกับการ
จัดรูปแบบสถานการณ์น่ารำคาญนั้นด้วย

- **ตัวอย่างเช่น, นี่คือรายการที่ซ้อนกันอีกตัวอย่าง**

  แต่คราวนี้มีย่อหน้าที่สอง

  - รายการเหล่านี้จะไม่มี `<p>` tags
  - เพราะพวกเขามีเพียงบรรทัดเดียวแต่ละรายการ

- **แต่ในรายการด้านบนที่สองระดับนี้, พวกเขาจะมี**

  นี่เป็นที่น่ารำคาญมากเพราะระยะห่างในย่อหน้านี้

  - ดังที่คุณเห็นที่นี่ เนื่องจากฉันได้เพิ่มบรรทัดที่สอง รายการนี้ตอนนี้มีแท็ก `<p>`

      นี่คือบรรทัดที่สองที่ฉันกล่าวถึงนะ

  - สุดท้ายนี้เป็นรายการอีกตัวที่ทำให้มันดูเหมือนรายการมากขึ้น

- รายการปิด แต่ไม่มีรายการที่ซ้อนกัน เพราะทำไมไม่?

และสุดท้ายนี้คือประโยคเพื่อปิดส่วนนี้

## มีองค์ประกอบอื่นๆ ที่เราต้องจัดรูปแบบ

ฉันเกือบลืมที่จะกล่าวถึงลิงก์ เช่น [ลิงก์นี้ไปที่เว็บไซต์ Tailwind CSS](https://tailwindcss.com)
เราเกือบทำให้มันเป็นสีน้ำเงิน แต่นั่นเป็นเรื่องเมื่อวาน
ดังนั้นเราเลือกให้มีสีเทาเข้ม รู้สึกที่เข้มงวด

เราได้รวมรูปแบบของตารางเข้าไปด้วย ลองดูดีมั้ย:

| นักมวยปล้ำ                 | ต้นกำเนิด       | ท่าไม้ตาย           |
| ----------------------- | ------------ | ------------------ |
| เบร็ท "เดอะฮิตแมน" ฮาร์ต  | คาลการี, AB  | นักแม่นปืน       |
| สโตน โคลด์ สตีฟ ออสติน | ออสติน, เท็กซัส  | สโตนโคลด์สตันเนอร์ |
| แรนดี้ ซาเวจ            | ซาราโซตา ฟลอริดา | ข้อศอกหล่น         |
| เวเดอร์                   | โบลเดอร์, โคโลราโด  | เวเดอร์บอมบ์         |
| เรเซอร์ รามอน             | Chuluota, FL | ขอบมีดโกน       |

เรายังต้องให้แน่ใจว่ารหัสแบบอินไลน์ดูดี อย่างเช่นถ้าฉันต้องการพูดถึงองค์ประกอบ
`<span>` หรือบอกข่าวดีเกี่ยวกับ `@tailwindcss/typography`

### บางครั้งฉันใช้ `code` ในหัวข้อบ้าง

แม้ว่ามันน่าจะเป็นไอเดียที่ไม่ดี และในอดีตฉันมีความยากลำบากในการทำให้มันดูดี
วิธี "_ห่อบล็อกโค้ดด้วยเครื่องหมายย้อนกลับ_" นี้ทำงานได้ดีจริง ๆ

อีกสิ่งหนึ่งที่ฉันเคยทำในอดีตคือใส่แท็ก `code` ภายในลิงก์, เช่น ถ้าฉันต้องการบอก
คุณเกี่ยวกับเกี่ยวกับ [`tailwindcss/docs`](https://github.com/tailwindcss/docs)
repository. ฉันไม่ค่อยชอบว่ามีขีดเส้นใต้ด้านล่างของเครื่องหมายย้อนกลับ แต่มันก็ยิ่งไปกว่าความบ้านิ่งที่จะหลีกเลี่ยงมัน

#### เรายังไม่ได้ใช้ `h4` จนถึงตอนนี้

แต่ตอนนี้เราใช้ `h4` แล้ว กรุณาอย่าใช้ `h5` หรือ `h6` ในเนื้อหาของคุณ Medium
รองรับเพียงสองระดับหัวเรื่องเพื่อเหตุผลหนึ่ง คุณคือสัตว์นะ ในความจริงฉันคิดเลือก
ใช้ `before` pseudo-element เพื่อตะโกนให้คุณฟังหากคุณใช้ `h5` หรือ `h6`

เราไม่ได้ทำการจัดรูปแบบใด ๆ ให้กับ `h4` ทันที เพราะ `h4` มีขนาดเล็กมากแล้ว ทำให้
มีขนาดเท่ากับข้อความในเนื้อหา. เราควรทำอะไรกับ `h5`, ทำให้มัน _เล็กกว่า_ ข้อความ
ในเนื้อหาหรือไม่? ไม่ขอบคุณ

### เรายังต้องคิดถึงหัวข้อที่ซ้อนกัน

#### ให้ตรวจสอบให้แน่ใจว่าเราไม่ทำให้มันผิดพลาดกับ `h4` ด้วย

ฮู้ว โชคดีครับถ้าเราได้ทำการจัดรูปแบบให้กับหัวข้อด้านบนข้อความนี้และดูดีมาก

เรามาเพิ่มย่อหน้าปิดที่นี่เพื่อให้สิ่งต่างๆ ลงท้ายด้วยบล็อกของข้อความที่มีขนาดเหมาะสม
ฉันไม่สามารถอธิบายได้ว่าทำไมฉันต้องการให้สิ่งต่าง ๆ จบลงมาแบบนี้ แต่ฉันต้องสงสัย
ว่าเป็นเพราะฉันคิดว่าสิ่งต่างๆ จะดูแปลกๆ หรือไม่สมดุลถ้ามีหัวข้อที่สุดท้ายใกล้กับจุดสิ้นสุดของเอกสาร

สิ่งที่ฉันเขียนที่นี่น่าจะเพียงพอยาวแล้ว แต่การเพิ่มประโยคสุดท้ายนี้ก็ไม่ได้เป็นผลกระทบใดๆ
